## What I actually found

None of the above. The notification stack was never regressed. It was a **buildout gap that has existed since Phase 12 first shipped**.

Architectural map of fire paths BEFORE this session:

| Action | Code path | Fires? |
|---|---|---|
| Public booking page → new booking | BookingPage.fireBookingConfirmation | ✓ |
| Therapist Schedule → + New Booking | BookingModal direct insert | ✗ |
| Therapist Schedule → Reschedule | BookingModal update | ✗ |
| Public booking page → save card + pay | capture-saved-card edge func | ✓ |
| CheckoutModal card-on-file | charge-card (no fire) | ✗ |
| CheckoutModal new card | save-card + charge-card (no fire) | ✗ |
| CheckoutModal payment link | create-payment-link (no fire) | ✗ |
| MarkAsPaidModal | direct insert | ✗ |
| Cancellation no-fee path | notify-booking-event | ✓ |
| Cancellation fee charged | charge-cancellation-fee | ✓ |
| No-show recorded | notify-booking-event | ✓ |
| Stripe Dashboard refund | stripe-refund-webhook (no fire) | ✗ |
| In-app refund | refund-session-payment (no fire) | ✗ |

7 gaps. All in code I or prior sessions wrote in Phases 12, 13, 14. Public booking page was the only consistently-firing path. Therapist-side surfaces never were.

Resend was not the issue: any rate-limited Resend call would still write a notification_log row with status='failed' or 'rate_limited', and there were zero rows for these events. The fire was never invoked, so Resend was never called.

## What shipped

Three commits, 45 min total, under the 75-min estimate.

### `e0827e1a` Phase 15.1: BookingModal fires send-booking-confirmation

Captured the inserted booking id via `.select('id').single()`, added a fire-and-forget `fireBookingConfirmation` helper mirroring BookingPage's pattern. Called after both create and reschedule paths. event_type param passed for future matrix differentiation; edge function currently ignores it but the contract is in place.

Reschedule path note: send-booking-confirmation reads fresh booking data on every invocation, so calling it after a reschedule sends 'your appointment is confirmed for [new date/time]' which is correct copy. A dedicated reschedule template is a future polish.

### `c87872ab` Phase 15.2: notify-payment-event + Checkout + MarkAsPaid

New edge function `notify-payment-event`:
- Accepts `{ session_payment_id }`
- Loads payment + booking + client + therapist
- Therapist fan-out: Bell + Email (with breakdown card) + SMS + Push
- Client fan-out: Email (receipt-style with Service/Method/Tip/Total) + SMS thank-you
- Logs every channel attempt to notification_log
- Defensive: skips if status != succeeded

Wired into three insert points:
1. CheckoutModal chargeCardOnFile success
2. CheckoutModal new-card success
3. MarkAsPaidModal save success

All three now use `.select('id').single()` to capture the row id, then fire-and-forget call.

NOT wired: CheckoutModal payment-link path. Fire should happen when the client COMPLETES the link, not when the therapist sends it. That's a separate webhook flow tied to stripe-payment-link-webhook. Deferred as Phase 15.4 if needed.

### `fcbf56dd` Phase 15.3: notify-refund-event + 3 refund paths

Added to notificationSpec.js:
- T14: refund_issued for therapist (bell + push + email)
- C16: refund_issued for client (email + SMS)

Compliance Dashboard matrix at /founder/notifications now renders these.

New edge function `notify-refund-event`:
- Accepts `{ session_payment_id, source? }`
- Same data load pattern as notify-payment-event
- Branches copy on `isStripe` (payment_method starts with 'stripe_')
  - Stripe: "amount returns to card in 5 to 10 days"
  - Offline: "remember to return the cash/Venmo/etc separately"
- Defensive: skips if status != refunded

Wired into three paths:
1. refund-session-payment OFFLINE path (in-app Cash/Venmo refund) → fire with source='in_app_offline'
2. refund-session-payment STRIPE path (in-app card refund) → fire with source='in_app_stripe'
3. stripe-refund-webhook (external Stripe Dashboard refund) → fire with source='webhook'

Idempotency design: webhook only fires if IT is the one flipping the row. If status was already 'refunded' (because in-app function fired first), the existing idempotency check returns early before the fire point. **One notification per refund regardless of initiator.**
