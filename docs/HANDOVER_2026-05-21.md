## Key design lessons from today (don't forget)

1. **Mobile-first or it doesn't ship.** Jackie's "third world" screenshot of the block panel exposed that the prose-style layout collapsed badly on mobile. From now on, imagine the 375px iPhone render before committing any UI change. Multiple recent commits skipped this and shipped broken-on-mobile layouts.

2. **The 60-day Schedule cap was an unexamined arbitrary number** from April 1 (commit `f9bf30495`). Set during early build-out, never a design choice. Real therapists like Jackie book a year out. Lesson: any cap that was set for "safety" early on should be revisited as real customer data accumulates.

3. **Silent auto-creation is dangerous.** Import flow was happy to auto-create 397 fake services + 397 memberships + 608 subscriptions from a single mis-mapped column. Pre-flight checks now catch this for both Service and Membership Plan columns. Pattern worth applying to any other silent-auto-create paths.

4. **Maria persona = collapsed advanced + clear defaults.** Old import UI showed 15 dropdowns. New UI shows 4 green checkmarks ("What we found in your file") + an "Advanced options" collapsible. Wall-of-dropdowns is the wrong UX for non-technical therapists, even with explanatory text.

5. **Don't use `type="number"` for therapist-facing inputs.** Buffer minutes input fought Jackie's typing (step constraint + parseInt fallback to default on empty string). Use `InlineSaveNumberInput` everywhere. Standing rule, applied today but worth reaffirming.

6. **Skipped vs Failed in imports** needed clearer distinction:
   - Skipped = deliberately didn't import (already exists, missing required, duplicate in file)
   - Failed = tried but couldn't (date parse error, DB error)
   - Both now download as CSV with `bm_reason` + `bm_details` columns

7. **Jackie diagnosed her own duplicate appointments** ("MassageBook splits packages into multiple rows"). The platform skipped 21 duplicates correctly and she figured it out without help. Trust the user's ability to interpret good error messages.

8. **A customer who flags 6+ bugs in one day = gold.** Treat Jackie like a partner. She caught: buffer-input bug, 60-day Schedule cap, mobile block panel, invisible full-day blocks, $0 services from import, "where are skipped rows". All real. All shipped fixes. Worth a thank-you gesture or testimonial ask once she's truly settled.

## Things NOT to repeat

- **Don't ship UI changes without imagining mobile render.** Multiple times today I shipped layout-via-flex that broke on 375px width. Jackie caught it. Going forward: when changing any UI, picture the iPhone width before committing.
- **Don't assume bookings render in Schedule because the data is there.** I had to debug "she can't see her appointments" only to find the 60-day cap. The query worked, the data was correct, the UI just filtered them out. Always check the render layer when data appears missing.
- **Don't refactor large flows mid-customer-use.** I considered shipping a full service-consolidation UI tonight while Jackie was actively using the platform. Bad idea. SQL on the backend is safer than mid-session UI churn.
