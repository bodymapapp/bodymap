Notifications.

The five things to remember

Design principles for the notification engine

  1. One engine, many templates. Every notification on the platform flows through a single delivery engine. Adding a new notification means writing a template, not a new code path. This is the discipline that prevents the 100-tests-per-notification trap.
  2. Test the engine once, then trust it. The engine is the only thing that needs heavy testing. Once the delivery pipeline works for receipts, it works for reminders, lapse nudges, and apology emails. Per-notification testing is reduced to "does this template render with this data?"
  3. Templates are content, not code. Every notification is defined as a row in a database table: event name, channels, conditions, and the text of the message. Adding a notification or changing copy is a content edit, not an engineering ticket.
  4. Channels are pluggable. SMS via Twilio, email via Resend, in-app via the bell drawer. Each one is a separate adapter. When SMS goes down, the engine logs the failure and tries email as a fallback (per the channel-priority field on the template).
  5. Every send is observable. Single events table logs who got what, when, on which channel, and whether delivery succeeded. One dashboard surfaces deliverability problems before clients call.
★ A guide for building, not buying

How to build the notification engine without losing your mind

A practical architecture for the notification system behind a platform like MyBodyMap. Designed to be built once, deployed quickly, and extended without ceremony. Written for the solo founder who is anxious about how many moving parts notifications appear to have.

Companion to the Money and Journey playbooks. Those describe what notifications should say and when they should fire. This one describes how to build the machine that does it.

Why this playbook exists

If you read Money and Journey, you saw a couple of dozen specific notifications, each with channel rules, suppression conditions, copy variants, and personality-of-the-relationship tuning. The natural fear: "I have to build and test each of these as a separate feature."

You don't. You build one notification engine, and every notification becomes a configuration of that engine. The first notification takes a long time to ship because you're building the engine. Notifications two through fifty take an hour each because you're filling out a template.

This playbook is the design of that engine, the order to build it in, and the test pyramid that keeps you sane.

The architecture in one diagram

┌───────────────────────────────────────────────────────────┐ │ APP CODE │ │ "client just no-showed" → emit('booking.no_show', ctx) │ └───────────────────────────────────────────────────────────┘ ↓ ┌───────────────────────────────────────────────────────────┐ │ NOTIFICATION ENGINE │ │ │ │ 1. Look up all templates where event = 'booking.no_show' │ │ 2. For each template, check audience + conditions │ │ 3. Resolve channel(s) and recipient(s) │ │ 4. Render template with context (Maria, $90, Tuesday) │ │ 5. Hand off to channel adapter (SMS / email / bell) │ │ 6. Log every attempt to notification_events table │ └───────────────────────────────────────────────────────────┘ ↓ ┌──────────────┬──────────────┬──────────────────────────┐ │ SMS adapter │ EMAIL adapter│ BELL adapter │ │ (Twilio) │ (Resend) │ (in-app, postgres) │ └──────────────┴──────────────┴──────────────────────────┘

The shape that makes this manageable: the app code only ever emits events. It never knows what notification will fire, who will receive it, or how. All of that is configuration in the engine. The same line of code that fires the no-show notification today will fire the new "we miss you" follow-up message we add next month, without a code change.

The six pieces of the engine

1Events table
The vocabulary the app code uses to describe what happened.
A small enum-like set of event names: booking.created, booking.confirmed, booking.no_show, booking.therapist_cancelled, client.lapsed, etc. Every part of the app emits one of these when something happens. The engine subscribes; nothing else.
2Templates table
Where every notification lives. The content, not the code.
One row per notification. Columns: event name, audience (therapist or client), channels (SMS, email, bell, in any combination), conditions (e.g. "only if client has fewer than 10 sessions"), subject line if email, body template with handlebars-style placeholders. Adding a notification is an INSERT, not a code change. Editing copy is an UPDATE.
3Renderer
Turns a template + context into actual text.
Takes the template body ("Hi {{client.first_name}}, we missed you on {{session.day}}") and the context object ({client: {first_name: 'Maria'}, session: {day: 'Tuesday'}}) and produces the final string. Same renderer for SMS and email. Plain-text rendering is trivial. HTML email rendering uses a layout template that wraps the body.
4Channel adapters
Three thin wrappers around Twilio, Resend, and the bell table.
Each adapter accepts a rendered message and a recipient identifier (phone number for SMS, email for email, user_id for bell). It returns success or a structured error. The engine doesn't know how Twilio works; the SMS adapter does. New channels (WhatsApp, push, etc.) are just new adapters.
5Events log
One table, every notification ever sent. The audit trail and the debugging surface.
Columns: event name, template id, recipient, channel, status, attempted_at, delivered_at, error if any. When Candice texts "Maria didn't get her reminder," you query this table by phone number and see exactly what fired and what happened.
6Suppression rules
The guardrails. What we never send.
Quiet hours (no SMS between 9pm and 8am client local time). Unsubscribed clients (one row per client per category). Bounce-then-block (3 hard bounces in a row → suppress that channel for this recipient). Rate limit (no more than 3 notifications to one client per day). These rules apply before the channel adapter runs.

The test pyramid that keeps you sane

The whole fear of "100 tests per notification" comes from imagining each notification as its own feature with its own integration test. Replace that mental model with three layers:

Layer 1
Engine integration tests
~30
Heavy. Covers the engine end-to-end: event in, message rendered, channel adapter called, log row created. Once these pass, you trust the engine. You write these once and rarely touch them.
Layer 2
Template render checks
~5 per template
Light. Did the template render? Are all placeholders filled? Is the SMS under 160 chars? Does the email subject have nothing weird? Snapshot tests per template, auto-generated.
Layer 3
Manual smoke test in staging
1 per template
Send the notification to your own phone and inbox in staging. Read it. Does it sound like a human wrote it? Ship if yes. This takes 2 minutes per template.

Math: engine integration tests take a week of work, once. Per-template work is roughly 15 minutes (writing the template) plus 5 minutes (smoke test). For 25 notifications across both playbooks, the total per-template effort is about 8 hours of work spread across days as the product matures. Not 100s of hours. Not anxiety-inducing.

The order to build it in

1
~2 days · highest leverage
The events table and the events log table
Both can be added to the schema in a single migration. The events log is just a postgres table. The events table is more of a TypeScript const than a database object, but documenting the vocabulary somewhere is critical. Without this, the rest of the engine has nothing to subscribe to.
2
~2 days
One channel adapter (start with SMS via Twilio)
Wire Twilio with a simple send function and a delivery webhook. Get it logging into the events log. This is the smallest end-to-end that proves the architecture: app emits event → adapter sends SMS → log records it.
3
~1 day
The renderer
Handlebars or any simple template library. Don't get fancy. Plain string templates with {{placeholders}} cover 95% of what we need. The other 5% (conditional content, pluralization) can be solved later.
4
~2 days
The templates table + the first three notifications
Build the table, build the admin UI to edit templates (later, when you have time; for v1, edit by SQL), and ship the first three notifications: booking confirmation, 48-hour reminder, post-session thank-you. Each one validates a different shape: instant, scheduled, and event-triggered-with-delay.
5
~1 day
Email adapter
Wire Resend. Same shape as the SMS adapter. The renderer already works; email is just a second output channel.
6
~1 day
The bell adapter (in-app)
Just a row in a notifications table the therapist's UI subscribes to. No external service. The simplest of the three adapters.
7
~1 day
Suppression rules
Quiet hours, unsubscribes, rate limits. These can be hard-coded constants for v1. Move them to a settings table later when therapists ask to customize.
8
ongoing · small per template
Add every other notification, one at a time
15 minutes per template. 5-minute smoke test. Ship. The engine is done; everything from here is just content.

Total engine build time: roughly 9-10 days of focused work for one developer. After that, adding a notification is a copy-and-paste operation that takes an afternoon for a batch of five.

The mental model

The platform is a vocabulary, not a list of features

The deepest mistake in building notification systems is treating each notification as a feature. The 48-hour reminder is a feature. The lapse nudge is a feature. The post-session thank-you is a feature. Under that mental model, you have 25 features to build, test, and maintain.

Replace it: the platform is a vocabulary of events the app emits, and a library of templates that respond to events. Adding a notification is adding a row to the library. It is not building, testing, or maintaining a feature.

Once this model lands, the anxiety lifts. The work in front of you is not 25 features. It is one engine and 25 content rows. The first half takes weeks; the second half takes hours.

You can ship one notification per day for a month and still have a moat

Once the engine is real, the cadence to follow is one or two notifications per day. Not all at once. Smoke-test each, get one therapist to vet the copy, ship, watch the events log, move on.

By the end of one month: 25 notifications shipped, each tested in real practice, copy iterated based on actual client reactions. Far ahead of any competitor who only built the basics.

The hardest part is writing the copy, not the code

The engine is 9 days of focused work. Once it exists, the differentiation is in the copy itself: tone, warmth, voice, timing. The two playbooks (Money and Journey) already define the framework for the copy. The work of writing each template is roughly 30 minutes of editorial effort per notification, multiplied by 25 notifications, which is 12.5 hours of work that happens over weeks, not a sprint.

This is the work that no competitor will do well. Most send technically-correct but emotionally-cold notifications. The opportunity is in the warmth of the copy, not the sophistication of the engine.
